# AI 使用说明

这份文档说明我在完成这个 AI Coding 作业时如何使用 AI，以及哪些设计判断是我自己做出的。

## 1. 我对问题的理解

这是一个企业内部将下游不同通知供应商进行聚合，并且将业务事件异步投递到供应商的系统。要实现的不是一个简单的http 透明代理，而是一个异步任务运行平台

- 下游多个通知供应商如何快速拓展，适配不同供应商的api 请求方式，请求参数，请求头，请求体等
- 如何保证投递的幂等性（要默认下游供应商都是非幂等的，需要我们在这个业务平台中实现幂等）
- 下游故障时如何处理，如何重试，如何保证投递的可靠性，如何尽可能保证投递的顺序性
- 第一版完成到什么程度，最小MVP 做到什么阶段
- 如何监控运维任务投递进度（做一个简单的监控面板）
- 一个业务通知api 调用量剧增，不能阻塞其他调用

所以我没有把系统做成“业务方传 URL/Header/Body，平台帮忙转发”的通用代理，而是把它定位为一个**供应商适配型通知平台**。

业务系统提交的是标准业务事件：

```json
{
  "provider_code": "crm",
  "event_type": "subscription_paid",
  "event_id": "local_test_001",
  "payload": {},
  "metadata": {}
}
```

平台负责根据 `provider_code + event_type` 找到对应 adapter，把内部业务事件转换成供应商需要的 HTTP 请求。这样供应商协议变化时，只需要改通知平台，不需要多个业务系统一起改。

## 2. 我如何规划这个项目

我一开始先把问题拆成几层：

1. 对外 API：业务系统如何提交通知，请求的api 是什么格式，如何将尽可能满足业务需求，并且保证幂等性
2. 供应商适配：不同供应商的 URL、Header、Body、认证方式如何封装，如何快速拓展适配
3. 异步投递：如何让业务请求和外部 HTTP 调用解耦；
4. 可靠性：失败、超时、重试、重复提交怎么处理，下游通知供应商一直故障如何解决。
5. 持久化：任务状态和投递尝试如何记录；
6. 运维能力：如何查看堆积、失败、重试、暂停供应商；
7. 交付验证：如何让别人可以本地启动、测试、看结果。

## 3. 整体架构与核心设计

最终架构是：

```text
业务系统/技术栈选择：
  -> FastAPI 提交通知/实现对外 API 和运维 API；
  -> PostgreSQL 持久化任务、供应商、投递尝试；
  -> Dramatiq actor 实现异步队列和重试调度；
  -> Redis broker/消息队列；
  -> worker 消费任务
  -> provider adapter 构造供应商 HTTP 请求
  -> httpx 投递到外部供应商
  -> Alembic：管理数据库迁移；
  -> Docker Compose：一键启动 API、worker、Postgres、Redis 和 mock vendor；
  -> pytest：覆盖 API、adapter、worker 和集成流程。
  -> github actions: 自动化测试，打包，部署

运维 Web
  -> FastAPI admin API  查看任务状态、供应商队列、投递尝试、失败原因
```

我选择这个组合，是因为它既能满足生产环境使用的基本要求，又能满足我们的基本需求和后续功能的拓展，又没有引入 Kafka、Celery 全量能力那种超出第一版的复杂度。

下面把每一项的选型理由和当时考虑过的替代方案列出来，方便理解每一个技术栈的取舍：


| 技术栈                | 为什么选它                                                                                          | 替代方案 / 没选的原因                                                                                                             |
| ------------------ | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **FastAPI**        | 类型注解 + Pydantic 自动校验，自带 OpenAPI 文档；原生支持 async，                                                 | Flask：没有原生 async，参数校验和文档要手写。Django + DRF：偏重，对纯 API 项目过度。                                                                 |
| **PostgreSQL**     | 投递任务、供应商、投递历史这种状态机数据需要事务、唯一约束（幂等 key）、JSONB（payload/metadata）和并发安全的 `UPDATE ... RETURNING`。    | 我个人使用pg 较多，直接支持生成环境，后续无需迁移，SQLite：不支持多 worker 进程并发写、JSON 表达力差，只适合 demo。MySQL：JSON 操作和并发 `UPDATE ... RETURNING` 不如 PG 顺手。 |
| **Dramatiq**       | 重试、退避、消息确认开箱即用；actor 模型语义清晰；按 queue 分流天然支持"一供应商一队列"；之前在生产环境用过，踩坑少。                             | RQ：太轻，无法满足复杂任务调度需求。Celery：功能完整但配置和概念偏重，                                                                                  |
| **Redis**          | Dramatiq 推荐 broker，部署最简单（一个容器），延迟低；同时可以未来扣到限流、缓存。                                              | RabbitMQ：可靠性更高但要单独运维。Kafka：第一版完全不需要它的吞吐能力。                                                                               |
| **httpx**          | 同时支持同步/异步、连接池、超时配置；和 FastAPI 同生态；测试有 `pytest-httpx` 可以拦截外部请求。                                  | `requests`：无 async、超时和连接池配置不够灵活。                                                                                         |
| **Alembic**        | 和 SQLAlchemy 同生态，迁移文件可 review/回滚；CI 里加一行 `alembic check` 即可检查一致性                               | 手写 SQL 脚本：无版本管理、无回滚、容易漏。                                                                                                 |
| **Docker Compose** | 一条 `docker compose up` 拉起 API、worker、Postgres、Redis、mock vendor，本地和 CI 用同一份 compose 文件，环境差异最小。 | 手动启动运维部署太复杂，也不需要k8s 这样重的                                                                                                 |
| **pytest**         | fixture 体系灵活，能很方便构造"假供应商响应 + 真 DB"的集成测试                                                        | `unittest`：写法啰嗦，fixture 不灵活。手写脚本：                                                                                        |
| **GitHub Actions** | 仓库自带、零额外运维；并发跑 lint/test/build/docker push 都顺手；GHCR 镜像和 Actions 串起来发布很方便。                      | 依托github 生态，更方便，根据不同仓库管理平台也可以选择gitlab ci                                                                                 |


## 4. AI 在哪些关键地方提供了帮助

第一，帮助拆解作业要求。

AI 帮我从 PDF 中提炼出评估重点：系统边界、复杂度管理、对 AI 输出的判断能力、真实工程经验。这个提醒让我没有只追求“功能多”，而是把重点放在为什么这样设计。

第二，帮助比较技术方案。

讨论中 AI 给过从轻到重的方案，比如：

- FastAPI + SQLite + 内置 worker；
- FastAPI + PostgreSQL + Redis + RQ；
- FastAPI + PostgreSQL + Redis + Dramatiq；

这些比较帮助我明确：这个作业需要体现真实异步投递和失败处理，但不需要上来就用最重的基础设施。最终我选择了 PostgreSQL + Redis + Dramatiq。

这里也体现了一次重要的判断修正：AI 最开始更偏向推荐轻量 MVP，例如 FastAPI 加 SQLite 或简单后台任务。这个方向实现快，但我认为它不够符合这次作业强调的“真实工程经验”。真实的通知投递系统通常需要持久化任务状态、独立 worker、消息队列、失败重试和运维可见性。因此我把技术选型调整为 FastAPI + PostgreSQL + Redis + Dramatiq，并补上 Docker Compose、pytest 和 React 运维页面，让项目做到一个基本可以使用的内部服务。

第三，帮助暴露可靠性问题。

AI 在讨论中不断提醒一些边界问题：

- 下游供应商可能非幂等；
- 请求发出但响应失败时，平台无法知道外部副作用是否发生，这种情况第一版不做过多设计，无成功返回的通知调用即认为失败，由后续重试机制处理；
- 消息队列可能重复投递；
- 失败任务不能只改状态，还要重新入队；
- adapter 转换失败不应该无限重试，因为这种不是外部原因，重试是不能解决问题的，需要人工介入处理。

第四，经过多轮的讨论技术点和实现的需求边界，然后让ai 生成spec 文档，我再次review spec 文档，进一步修改使其符合我的需求，最终形成spec 文档。

第五，ai review spec 文档，然后按照spec 文档生成开发计划，按照开发计划生成代码。

最后让 AI 自动化测试功能、review 代码，修复一些和 spec 不一致以及错误的逻辑，更新 README 文档，我再手动测试交付验证，完成项目交付。

## 5. AI 给过但我没有采纳的建议

### 5.1 没有采纳 SQLite + 内置 worker

这个方案实现最快，但更像 demo。它不能很好体现消息队列解耦、worker 独立部署、供应商失败隔离和任务状态持久化。因此我改为 PostgreSQL + Redis + Dramatiq。

### 5.2 没有采纳 RQ、Celery 或 Kafka

RQ 比较简单，但 actor 模型和中间件能力不如 Dramatiq 适合这个场景。我之前在项目生产环境中使用的Dramatiq，更加熟悉，也能满足我们的这个项目需求，所以倾向于选择这个。Celery 功能完整，但配置和概念偏重。Kafka 更适合大规模事件流，不是这个作业第一版必须引入的复杂度。

所以我选择 Dramatiq：比 RQ 更适合做带重试的任务系统，比 Celery 更轻。

### 5.3 没有采纳 `uncertain` 状态

AI 曾建议：如果请求可能已经发出但响应未知，可以进入 `uncertain` 状态，由人工确认后再处理。

这个设计更严谨，但会引入额外状态、人工流程和运维复杂度。第一版我选择更简单的规则：没有明确成功就按失败处理并重试，同时在文档里说明下游非幂等风险。不过多设计增加系统复杂度 

### 5.4 没有采纳自动熔断和自动探测恢复

AI 曾建议做自动熔断：某供应商连续失败就自动暂停，并用探测任务自动恢复。

我没有采纳，因为这需要定义窗口、阈值、恢复策略、人工暂停优先级，而且探测如果使用真实任务，还会加大非幂等风险。第一版保留人工暂停/恢复，把自动熔断作为未来演进。

### 5.5 没有采纳供应商热更新和请求模板 DSL

AI 讨论过通过页面配置供应商 endpoint、headers、body 模板，甚至做请求模板 DSL。

我没有采纳，因为供应商协议包含认证、签名、字段转换、密钥脱敏、回滚和测试问题。对第一版来说，typed Python adapter 更容易 review、测试和维护。

### 5.6 没有采纳复杂重试配置

我主动提出错误后的退避策略，但是ai建议供应商配置最大次数、退避数组、retry policy 等。

我认为第一版配置面越大，越容易出错，也会分散重点。最终只保留 `DEFAULT_MAX_ATTEMPTS` 一个环境变量，退避策略由系统内置。要不然配置上手太复杂，先满足能用的需求

### 5.7 没有采纳暂停后冻结队列内任务

AI 曾建议供应商暂停后，队列里已经存在的任务也停止投递，等恢复后再批量处理。

我没有采纳。这个设计需要新状态和恢复流程。第一版暂停只阻止新任务进入；已经入队的任务继续按失败、重试、failed 的流程走，行为更简单。也更容易理解

## 6. 我做出的关键决策与原因

### 6.1 选择“供应商适配型平台”，不是透明 HTTP 代理

这是我认为最重要的产品形态决策。

如果让业务系统直接提交供应商 URL、Header 和 Body，通知平台会变成一个通用转发器，业务系统仍然要理解每个供应商协议。这样供应商一变，多个业务系统都要跟着改。

我选择让业务系统提交标准业务事件，由平台 adapter 负责转换供应商请求。这让系统边界更清楚：业务系统负责表达“发生了什么”，通知平台负责“如何通知供应商”。

### 6.2 采用至少一次投递，不承诺 exactly-once

外部 HTTP API 尤其是非幂等 API，平台无法单方面保证 exactly-once。比如请求已经发出，但响应超时，平台无法知道供应商是否已经产生副作用。

所以我选择至少一次投递。符合“尽可能可靠送达”的目标。重复副作用风险通过有限重试、投递记录、人工运维和文档说明来管理。

### 6.3 平台内部必须幂等

虽然不能保证外部供应商 exactly-once，但平台内部必须尽量幂等：

- `provider_code + event_type + event_id` 唯一；
- 完全重复提交返回已有任务，不重复创建；
- 相同 key 但不同 payload/metadata 返回 `409 idempotency_conflict`；
- worker 只允许 `pending/retrying` 任务进入 `delivering`；
- 已成功任务不会被自动重复投递。

这是异步系统的基本底线。

### 6.4 按供应商队列隔离

我选择每个示例供应商有自己的队列，例如：

```text
notifications_crm
notifications_ads
notifications_inventory
```

原因是一个供应商长期故障时，不应该阻塞其他供应商。运维页面也应该能按供应商看到 pending、retrying、failed，而不是只看到一个全局总数。

如果未来供应商很多，可以从“一供应商一队列”演进为队列池，例如 critical、standard、bulk、quarantine。

### 6.5 使用数据库记录业务状态和投递历史

Dramatiq 负责消息级重试，但业务状态不能只依赖队列系统。

我选择把 notification 状态和每次 delivery attempt 都写入 PostgreSQL。这样运维页面能看到每次尝试的时间、URL、响应状态、错误类型和错误信息，也方便审计和排查。不过后续数据量大的时候需要使用缓存机制，避免因数据库带来吞吐量瓶颈。

### 6.7 我提出做轻量运维页面，ai 帮我实现

题目没有要求必须做前端，但我认为可靠投递系统必须能被观察和操作。只提供 API，不够符合真实运维体验。

所以我做了一个轻量 `/ops` 页面，支持：

- 按供应商查看队列任务状态；
- 查看通知列表和详情；
- 查看投递尝试历史；
- 暂停/恢复供应商；
- 人工重试 failed 任务；
- 手动刷新列表。
可以做一些简单的运维查看监控操作

### 6.8 供应商扩展走代码 adapter，并写 AI-ready 文档

暂时没有做页面新增供应商，而是提供 `docs/provider-onboarding.md`。

原因是供应商接入本质上是协议适配，往往涉及字段转换、认证、签名、成功码判断和测试。用代码 adapter 更可控，也更适合让 AI coding 工具基于文档生成新供应商接入代码和测试。

当前示例覆盖了 API Key、Bearer Token 和不同 payload 映射方式。query 参数、form body、签名 header 等也可以在 adapter 中扩展。

### 6.9 Docker Compose 自动迁移数据库

我选择让 Compose 启动时通过一次性 `db-setup` 服务自动执行 Alembic migration 和供应商 seed。

原因很简单：开发者本地启动时，不应该再记一串手动迁移命令。`docker compose up --build` 应该尽可能接近可用状态。

### 6.10 ai 最开始直接做了两个 docker-compose 文件

AI 最开始直接做了两个 docker-compose 文件，一个是开发环境，一个是生产环境，我主动拒绝了，这里会大大增加使用者成本，应该只有一个 docker-compose 文件，开发环境用 `docker compose up --build`，生产环境用 `docker compose up -d --no-build`（直接使用上游 GitHub 打包的镜像）即可。

### 6.11 容器运行时不用 `uv run`

构建镜像时已经执行 `uv sync --frozen --no-dev`。如果容器启动时继续使用 `uv run`，它可能重新同步依赖，导致启动阶段下载 dev 包。

我把运行命令改成直接使用 `.venv/bin/python`、`.venv/bin/uvicorn` 和 `.venv/bin/alembic`，让容器启动更快、更可预测。

### 6.12 用 CI 和统一镜像证明可交付

我加入了 GitHub Actions：

- 后端 lint；
- pytest；
- Alembic migration；
- 前端 build；
- Docker 镜像构建。

API、worker、mock vendor 使用同一个镜像，通过不同 command 启动。这样本地测试和远程镜像使用的是同一套交付物，减少环境差异。

## 总结

在这个项目中，我没有选择最简单的 demo，也没有选择最复杂的架构，而是在两者之间找了一个适合第一版交付的范围：

- 有真实工程中的队列、worker、数据库、迁移、测试、Docker 和 CI；
- 明确承认外部非幂等 API 的边界，在此基础上本服务保证上游调用幂等性；
- 用 adapter 管理供应商差异，而不是把复杂度推给业务系统；
- 提供运维可见性和人工兜底，
- 对 AI 给出的建议逐项判断，保留有价值的部分，拒绝超出第一版范围的设计。
